Card parent/child relations are now reachable from the CLI and the MCP server, not only the TUI. A new top-level `kanban relation` subcommand exposes `add`, `remove`, `parents`, and `children`; four matching MCP tools (`tool_set_card_parent`, `tool_remove_card_parent`, `tool_list_card_parents`, `tool_list_card_children`) cover the same surface for LLM clients.

```
kanban relation add KAN-5 KAN-7              # KAN-7 is now a subtask of KAN-5
kanban relation add KAN-5 KAN-7 KAN-8 KAN-9  # attach several children in one call
kanban relation children KAN-5               # list direct children
kanban relation parents KAN-7                # list direct parents
kanban relation remove KAN-5 KAN-7
```

Under the hood this lands several structural refactors that the new surfaces depend on:

- **Graph primitives in `kanban-core`.** A small trait taxonomy where each trait has one cohesive purpose:
  - `Graph`: minimal direction-agnostic edge contract: `add_edge`, `remove_edge`, `contains_edge`.
  - `Directed: Graph`: directed traversal vocabulary: `outgoing`, `incoming`.
  - `Undirected: Graph`: undirected traversal vocabulary: `neighbors`.
  - `Cascadable: Graph<NodeId=Uuid>`: node-level cascade mutations: `archive_node`, `unarchive_node`, `remove_node`.
  - `EdgeSet: Graph<NodeId=Uuid>`: set-like read access over edges, with stdlib-aligned names: `len`, `active_len`, `contains`, `is_empty` (now defined as "no active edges" so an all-archived graph reports empty).
  - `GraphError` enum: `Cycle`, `SelfReference`, `EdgeNotFound`.

  Two concrete implementations: `DagGraph` implements `Directed + Cascadable + EdgeSet` (cycle-rejecting), `UndirectedGraph` implements `Undirected + Cascadable + EdgeSet` (cycle-permitting). Both wrap a single `EdgeStore` type. `Edge` no longer carries a relation-kind type parameter; the kind lives at the sub-graph layer, so freshly saved files have no `edge_type` field at all and the on-disk byte shape matches what the migration produces. Direction-specific vocabulary lives only on the matching subtype trait; there is no way to ask an `UndirectedGraph` for directed semantics, the type system refuses. Each subtype trait is the *only* access path to its vocabulary (no inherent shadow methods, no inherent cascade duplicates), so callers bring the trait into scope and the trait stays load-bearing rather than decorative. `Cascadable` and `EdgeSet` are deliberately separate: a read-only consumer of edge counts no longer carries mutation authority via its trait choice. Filter accessors on `EdgeStore` (`outgoing`, `incoming`, `outgoing_active`, `incoming_active`, `active_edges`) return `impl Iterator<Item = &Edge>` rather than `Vec<&Edge>`, so the borrow chain stays allocation-free.

- **No `insert_raw_edge` escape hatch.** The previous load-time helper that bypassed structural validation is gone. Persistence backends (SQLite `rows_to_graph`, JSON `Deserialize` on `DagGraph` / `UndirectedGraph`) and test fixtures all route through validating constructors: `DependencyGraph::from_validated_edges(Vec<(CardEdgeType, Edge)>)` or sub-graph-level `add_edge_with_metadata`. A hand-edited or corrupt file with a cycle or self-loop now fails at load instead of silently rehydrating into an invariant-violating graph. Active-edge cycles are rejected; archived edges don't constrain the active DAG and can complete cycles through inactive paths.

- **Split `DependencyGraph` into three sub-graphs.** Replaces the single edge-type-tagged `CardDependencyGraph` and the `CardGraphExt` god-trait with three structurally honest sub-graphs:
  - `parent_child: DagGraph` (cycle + self-ref rejected)
  - `blocks:       DagGraph` (cycle + self-ref rejected)
  - `relates:      UndirectedGraph` (self-ref rejected, cycles permitted)

  Convenience methods (`set_parent`, `set_block`, `relate`, `parents`, `children`, `blockers`, `blocked`, `related`, `ancestors`, `descendants`, `can_start`, ...) live on `DependencyGraph` and delegate. Cross-cutting cascades split by capability: node-level mutations (`archive_node`, `unarchive_node`, `remove_node`, `disconnect`) iterate over `[&mut dyn Cascadable; 3]`; edge-level reads (`len`, `active_len`, `contains`) iterate over `[&dyn EdgeSet; 3]`. Both come from private helpers rather than hand-listing fields, so adding a fourth component graph only updates the helpers, not every cascade method. `disconnect`'s docstring now matches what the code does: it removes the specific oriented edge between two nodes (a→b for directed sub-graphs, either orientation for the undirected one); it does not strip both orientations. The old `blocked_by` (which paradoxically returned outgoing edges) is gone; the public surface is `blocked` (outgoing) / `blockers` (incoming) with names that match what they return.

- **Persistence routes through `DependencyGraph`.** A single per-kind accessor, `edges_of(kind: CardEdgeType) -> &[Edge]`, plus `edges_by_kind() -> impl Iterator<Item = (CardEdgeType, &Edge)>` and the validating `from_validated_edges` constructor, are the only public seams persistence backends and test fixtures use. The sub-graph fields are `pub(crate)`; no external caller reaches into `graph.parent_child` / `graph.blocks` / `graph.relates` directly.

- **`GraphOperations` returns `Vec<Uuid>`.** The trait surface deals in raw node ids; CLI / MCP / TUI surfaces resolve to display data themselves at their own boundary. One canonical parent/child spelling (`set_parent(child, parent)` / `remove_parent(child, parent)`) replaces the earlier two-alias surface — the alternative spellings were dropped to keep call sites consistent across surfaces. The trait has no `KanbanOperations` supertrait: graph topology is a separate concern. All three wrapper contexts (`CliContext`, `McpContext`, `TuiContext`) write the impl by hand because `TuiContext` needs `with_flush(...)` wrapping that a uniform macro cannot express; for two trivial passthrough contexts, a hand-written impl below that threshold is the right trade-off. Mutating calls go through `KanbanContext::add_card_edge` / `remove_card_edge`, which reject unknown card ids up front; the same `require_card_exists` guard now also runs on `list_card_edges_from` / `list_card_edges_to` so the read and write contracts are symmetric.

- **JSON format V6 with split-graph migration.** On-disk dependency graphs change shape from `graph.cards.edges: [{ edge_type, ... }]` to `graph.{ parent_child, blocks, relates }.edges: [{ ... }]`. Existing files at V1..V5 are auto-migrated on load through the appropriate chain of legacy steps and the new split-graph step. Migration aborts loudly on unknown or missing `edge_type` rather than silently dropping data, and the `edge_type` field is removed entirely (no `null`-padding); since `Edge` no longer has an `edge_type` field at all, freshly saved V6 files match the migration's shape byte-for-byte. Pre-V6 starting points (V3, V4, V5) get a `.v{N}.backup` written before the split-graph step and removed on successful migration, so an upgrade can be rolled back. Migration and sync paths both use `AtomicWriter` with a unique random temp file (`NamedTempFile::new_in(parent)`), closing the collision window that `path.with_extension("tmp")` left open.

- **SQLite hardened against corrupt edge_type values.** The `card_edges.edge_type` column now carries `CHECK (edge_type IN ('ParentOf', 'Blocks', 'RelatesTo'))`. New databases pick it up from the schema; existing databases are rebuilt one-off on open to add it. The write path uses `serde_json::to_value` to render enum variants symmetric with the `p_enum` read path, dropping the previous `format!("{:?}", kind)` coupling to the `Debug` derive.

- **Typed error boundaries.** New `KanbanCliError` (Domain / Message / Io / Serialization / Anyhow) and `KanbanMcpError` (Domain / Resolution / InvalidParam) types wrap `KanbanError` and let handlers thread every failure through `?` uniformly, with the boundary converting to the JSON `CliResponse` envelope (CLI) or `rmcp::ErrorData` (MCP). `KanbanCliError::Message { hint }` covers handler-built enrichment of anonymous domain errors; identifier-resolution failures flow through `Domain` directly so the structured `DomainError::NotFoundByName` / `Ambiguous` variants stay introspectable. The MCP `locked_read` / `locked_write` helpers are generic over `E: Into<McpError>` so typed closures plug in without per-handler conversion boilerplate. The relation surface is migrated to the new types now; migrating the remaining handlers is tracked as a separate ticket.

- **Shared parent-relation messages.** `kanban_domain::dependencies::messages` holds the formatters CLI and MCP both consume — cycle / self-reference / edge-not-found messages name both sides of the offending edge using the user's raw identifiers. The two surfaces now produce identical wording for the same failure. Both `enrich_*` helpers match exhaustively on `DependencyError` so a new variant fails to compile until the maintainer handles it, rather than silently slipping through an else branch.

Cycle detection and self-reference rejection from the underlying graph implementations flow up unchanged to every surface. Cross-board parent/child is permitted, matching the prior TUI behavior.

- **`DependencyCommand` collapsed to typed `AddEdge` / `RemoveEdge`.** Six per-kind structs (`AddBlocks`, `RemoveBlocks`, `AddRelatesTo`, `RemoveRelatesTo`, `SetParent`, `RemoveParent`) collapse into two, one per operation, each carrying `kind: CardEdgeType` so the sub-graph is selected at execute time. The on-disk JSON shape (`AddEdge` / `RemoveEdge` / `Remove` / `CreateSubcard` variant names plus their `kind` tags) is now pinned by a serialize round-trip test so SQLite command-log persistence (the schema exists but the writer is dormant) can treat it as a backwards-compatibility contract when it lands.

- **`CardEdgeType::ParentOf` renamed to `CardEdgeType::Spawns`.** The previous name was a noun-of-noun form inconsistent with the verb-shaped siblings `Blocks` and `RelatesTo`. Verb candidates that lose ancestry (`Contains`, `Owns`, `Includes`) were rejected; `Spawns` keeps the verb pattern and the transitive ancestry signal (A spawns B spawns C means A is an ancestor of C). The user-facing API stays parent/child (`set_parent`, `parents`, `children`) — that's how the relationship reads from either side. Migration handles the on-disk rename: SQLite databases get a one-time table rebuild that renames `'ParentOf'` rows to `'Spawns'` and tightens the `CHECK` constraint; pre-V6 JSON files continue to be recognised by the legacy-string-to-sub-graph routing in the split-graph migration.
